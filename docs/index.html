Possible project topics:

  • XSS:
    • Develop a tool that tries to understand which filters are used in a blacklist-based system by sending multiple requests, progressively approximating the filter by observing what gets inserted into the page (with as few requests as possible).
    • Build a system that learns from past XSS attacks performed against it and progressively adapts its filters (making them as general as possible without blocking legitimate input).
    • Develop a set of challenges with poorly designed whitelists that leave too much room for exploitation (they can become very complex quickly: it is important that there is always a reasonable way to solve them).
  • SQLi:
    • Tool that, given the ability to perform queries, determines (Project A: type and version; Project B: full schema) via SQL injection by attempting multiple techniques (union-based, error-based, time-based) and adapting its dialect based on the responses it receives.
    • (Simpler) Tool that analyzes code to find SQL injection vulnerabilities and proposes versions using prepared statements to prevent them.
    • In the challenges proposed in the lab there are many unintended solves: find as many as possible, list and analyze them, and propose ways to preserve the challenge (you cannot eliminate SQL injection using prepared statements, otherwise the challenge would no longer exist!) while preventing as many unintended solves as possible. If an unintended solve cannot be fixed or mitigated in any way, explain why.
  • CSRF:
    • Create a module to integrate into the platform where brute force is required to guess the token.
    • Create a module to integrate into the platform where defenses rely on SameSite and/or Referer headers (take inspiration from PortSwigger).
    • Additionally: develop a set of challenges where the goal is to convince an LLM that a request is legitimate (think Gandalf-style challenges) so that the bot executes or visits it.
  • Authentication / Protocols / Challenge-response:
    • Starting from the protocols studied in the lab, think of measures to make them more secure, introduce them one at a time, and evaluate which attacks are still possible (all accompanied by the creation of new challenges).
    • Improve the existing labs to include more robust state management and allow participants A and B to correctly complete the protocol.

CTF Platform

The platform is available at CTF-Platform-Web-and-Mobile-Systems-Security .

Prerequisites:

  • A Python interpreter (suggested Python 3.8+)
  • The Python packages fastapi and uvicorn
  • Docker and Docker Compose (see FAQ for docker compose vs docker-compose)

System Requirements:

  • Ports 8000–8100 should be available.
  • WARNING: The system is not guaranteed to be secure. Use it only on localhost.

How to Start It:

Enter the project folder and run:

make

Alternatively:

python3 ./main.py

On Windows:

python3 main-windows.py

You may need WSL installed.

Once started, access the platform at: http://localhost:8000/

How to Use It:

After opening the platform, you will see a list of containers.

Rule: 1 container = 1 laboratory = 1 port = 1 card

Each card provides the following actions:

  • Start - starts the container
  • Stop - stops the container
  • Build - builds the image
  • Logs - shows container logs
  • Open - opens the associated web service
  • Delete container - removes the container
  • Full reset - removes both container and image

Quick usage workflow:

  1. Press Start (auto-builds if needed)
  2. Click the web icon to access the lab
  3. When finished, press Full reset

How to Stop It:

CTRL + C

FAQ:

Docker compose vs docker-compose error:
Edit main.py (line 31 and others) and replace the command with the one supported by your system.

Docker permission issues:
Ensure your user is in the docker group. Then restart your shell and try again.

Nothing works:
Open an issue in the GitHub repository with a detailed description of the problem and any error messages you receive.
